# Изучение основ Java для Backend (ООП, Коллекции, Исключения, Дата/Время, Многопоточность)

Этот репозиторий содержит практические работы по повторению ключевых концепций Java, необходимых для уверенного перехода к разработке на Spring Boot. Проект объединяет в себе две основные части: проектирование бизнес-моделей на принципах ООП и создание отказоустойчивого, потокобезопасного реестра задач.

---

## 🚀 Что я изучил и применил в коде

### 1. Продвинутая работа с Коллекциями и Потокобезопасность

* **`ConcurrentHashMap` вместо `HashMap`:** В классе `TaskRegistry` для хранения задач была выбрана потокобезопасная ассоциативная замена — `ConcurrentHashMap<Long, Task>`. Это позволило избежать порчи структуры данных и падения приложения при одновременной записи из множества параллельных потоков без использования «тяжелых» блокировок (`synchronized`).
* **Динамические списки:** Метод `getAllTasks()` собирает и возвращает названия всех зарегистрированных задач в виде `ArrayList`, что является стандартным способом передачи коллекций данных между слоями приложения.

### 2. Проектирование иерархии исключений (Exceptions)

* **Кастомные Unchecked-исключения:** Создано собственное исключение `TaskNotFoundException`, наследующееся от `RuntimeException`.
* **Чистота сигнатур в Backend:** Отказ от checked-исключений (наследуемых от `Exception`) позволил избавить методы `getTask()` и `getMinutesSinceCreation()` от захламляющих код ключевых слов `throws`. Исключение выбрасывается вглубь логики, когда задача по указанному `ID` не найдена, и гибко перехватывается на самом верхнем уровне приложения (`Main`).

### 3. Современное API для работы с датой и временем (`java.time`)

* **Автоматическая фиксация времени:** Внутренний класс `Task` инкапсулирует поле `createdAt` с типом `LocalDateTime`. Время создания фиксируется автоматически с помощью `LocalDateTime.now()` прямо в момент вызова конструктора.
* **Расчет временных интервалов:** Реализован метод `getMinutesSinceCreation()`, высчитывающий точную разницу в минутах между моментом создания задачи и текущим временем с помощью `ChronoUnit.MINUTES.between()`.

### 4. Многопоточность и предотвращение Race Conditions (Состояний гонки)

* **Пулы потоков (`ExecutorService`):** В классе `Main` развернут пул потоков фиксированного размера `Executors.newFixedThreadPool(10)` для симуляции высокой конкурентной нагрузки на сервис.
* **Параллельная нагрузка:** Инициирован одновременный запуск 10 рабочих потоков, каждый из которых отправляет по 1000 запросов на добавление уникальных задач (`Task-ID`) в реестр.
* **Управление жизненным циклом потоков:** Использованы методы `executorService.shutdown()` и `executorService.awaitTermination()` для корректной остановки пула и ожидания завершения всех параллельных операций перед финальной проверкой данных.
* **Тестирование на Race Condition:** В коде реализована сверка ожидаемого количества задач (10 000) с фактическим размером коллекции. Благодаря правильному выбору `ConcurrentHashMap`, тест подтверждает успех выполнения без единой потери данных.

### 5. Фундамент ООП и Разделение ответственности (SOLID)

* **Абстрактные классы:** Создан базовый класс `Shape` с абстрактным методом `calculateArea()`, определяющий общий контракт для всех геометрических фигур.
* **Инкапсуляция и Наследование:** Классы `Circle` и `Rectangle` наследуют общее поведение, скрывая свои внутренние параметры (`radius`, `width`, `height`) за приватными модификаторами доступа.
* **Интерфейсы (Interface Segregation):** Выделен отдельный интерфейс `Drawable` с методом `draw()`. Это позволило разделить математическую сущность фигуры (`Shape`) и ее визуальное представление в консоли (ASCII-арт).
* **Полиморфизм времени выполнения:** В логике обработки фигур применяется динамическая диспетчеризация методов: программа итерируется по списку абстрактных `Shape`, вычисляя индивидуальные площади, и на лету проверяет поддержку интерфейса через `instanceof Drawable` для отрисовки графики.

## 🎯 Архитектурная связь со Spring Boot

Решения, отработанные в данных кодах, являются прямым аналогом того, как функционирует промышленный Backend:

1. **Многопоточность по умолчанию:** Каждый входящий HTTP-запрос в Spring Boot обрабатывается в отдельном потоке из пула веб-сервера (Tomcat). Написанный многопоточный тест доказывает готовность логики к конкурентной среде.
2. **Аналог Базы Данных:** Связка `ConcurrentHashMap` и кастомного класса `TaskRegistry` — это прототип слоя **Repository** (хранилища данных).
3. **Централизованная обработка ошибок:** Использование `RuntimeException` (`TaskNotFoundException`) подготавливает архитектуру к глобальному перехвату ошибок с помощью аннотации `@ControllerAdvice / @ExceptionHandler` в Spring, избавляя код от рутинных блоков `try-catch` на каждом шагу.
